# Feature Specification: Database Backup

**Feature Branch**: `[012-database-backup]`  
**Created**: 2026-04-29  
**Status**: Draft  
**Input**: User description: "增加数据库定期备份功能，每天15:30分，备份当前数据库文件。备份的数据库文件保留3天，超过3天的自动清理掉" + "增加手动备份功能，在设置页面增加'备份数据库'按钮，点击后执行数据库备份"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Automatic Daily Database Backup (Priority: P1)

系统每天在固定时间（15:30）自动备份数据库文件到同一目录，保留最近3天的备份并自动清理过期文件。

**Why this priority**: 数据备份是保护用户投资的核心功能，防止因意外情况导致的数据丢失。

**Independent Test**: 等待到达15:30，验证备份文件是否正确创建，检查过期备份是否被清理。

**Acceptance Scenarios**:

1. **Given** 系统正在运行，**When** 时间到达15:30，**Then** 系统在数据库文件所在目录创建带日期后缀的备份文件
2. **Given** 存在超过3天的备份文件，**When** 新备份完成后，**Then** 系统自动删除超过3天的旧备份文件
3. **Given** 备份过程中发生错误，**When** 备份失败时，**Then** 系统在日志文件中记录错误信息但不影响主程序运行

---

### User Story 2 - Manual Database Backup from Settings (Priority: P1)

用户在设置页面点击“备份数据库”按钮后，系统立即执行一次数据库备份，备份文件与自动备份使用相同的命名规则和存储位置。

**Why this priority**: 允许用户在重要操作前主动备份数据，或在自动备份间隔期间进行额外保护。

**Independent Test**: 打开设置页面，点击“备份数据库”按钮，验证备份文件立即创建，检查日志中的成功/失败消息。

**Acceptance Scenarios**:

1. **Given** 用户在设置页面，**When** 点击“备份数据库”按钮，**Then** 系统立即执行备份并在数据库目录创建带日期后缀的备份文件
2. **Given** 用户点击备份按钮，**When** 备份成功完成，**Then** 系统显示成功提示并记录日志
3. **Given** 用户点击备份按钮，**When** 备份失败时，**Then** 系统显示错误提示并记录详细错误信息到日志
4. **Given** 用户快速多次点击备份按钮，**When** 第一次备份未完成时再次点击，**Then** 系统忽略后续点击或显示“备份进行中”提示

---

### Edge Cases

- 如果15:30时系统未运行，下次启动时是否补做备份？（假设：不补做，只在运行时定时执行）
- 如果备份文件损坏或无法读取，如何处理？（假设：记录错误日志，继续清理流程）
- 如果磁盘空间不足导致备份失败，如何通知用户？（假设：记录错误日志，不中断主程序）
- 如果在备份过程中程序崩溃，如何处理部分完成的备份文件？（假设：使用临时文件机制，确保原子性）
- 时区变化或系统时间调整对定时任务的影响？（假设：使用系统本地时间）
- 如果同一天执行多次备份（自动+手动），如何处理命名冲突？（假设：追加序号如_001、_002）
- 如果用户在自动备份进行中点击手动备份按钮，如何处理？（假设：显示“备份进行中”提示或忽略点击）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST 每天在15:30自动执行数据库备份任务
- **FR-002**: System MUST 将备份文件保存到与当前数据库文件相同的目录下
- **FR-003**: System MUST 为每个备份文件使用日期后缀命名格式（例如：database_YYYYMMDD.db）
- **FR-004**: System MUST 保留最近3天的备份文件，自动删除超过3天的备份
- **FR-005**: System MUST 在每次备份完成后执行清理检查，删除过期备份
- **FR-006**: System MUST 在备份失败时将错误信息写入日志文件，不影响主程序正常运行
- **FR-007**: System MUST 确保同一时间只有一个备份任务在执行
- **FR-008**: System MUST 在设置页面提供“备份数据库”按钮供用户手动触发备份
- **FR-009**: System MUST 在用户点击手动备份按钮后立即执行备份操作
- **FR-010**: System MUST 在手动备份成功时向用户显示成功提示
- **FR-011**: System MUST 在手动备份失败时向用户显示错误提示并记录详细错误到日志
- **FR-012**: System MUST 防止用户在备份进行中重复触发动作（通过禁用按钮或忽略点击）

### Key Entities *(include if feature involves data)*

- **Backup File**: 数据库备份文件，文件名包含日期后缀（YYYYMMDD格式）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 系统能够在每天15:30±1分钟内成功执行数据库备份
- **SC-002**: 备份文件能够在3天后被自动清理，不会有超过3天的备份文件残留
- **SC-003**: 备份过程不影响主程序的正常运行
- **SC-004**: 备份失败时错误信息正确写入日志文件
- **SC-005**: 用户点击“备份数据库”按钮后，备份在5秒内开始执行
- **SC-006**: 手动备份成功率达到99%以上（排除磁盘空间不足等外部因素）
- **SC-007**: 用户在备份进行中再次点击按钮时，系统能够有效防止重复执行

## Assumptions

- 用户使用本地时区，系统使用本地时间进行定时任务调度
- 数据库文件在备份时可以安全复制（不需要停止服务）
- 备份文件与数据库文件位于同一目录，便于管理和查找
- 备份文件采用日期后缀命名（YYYYMMDD格式），同一天多次备份会追加序号（_001、_002）
- 程序在15:30时处于运行状态才能执行定时备份（不补做错过的备份）
- 备份采用文件复制方式，不涉及复杂的数据库导出导入流程
- 用户了解备份文件的位置和命名规则，可以手动访问备份目录
- 设置页面已存在或可以轻松添加新按钮
- 用户界面能够显示成功/失败提示（使用现有的Toast通知机制）
