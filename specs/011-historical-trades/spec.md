# Feature Specification: 历史开仓记录

**Feature Branch**: `012-historical-trades`
**Created**: 2026-04-23
**Status**: Draft
**Input**: User description: "增加历史开仓记录页面，展示历史交易中，每只股票在从持股数0开始首次买入，到清仓交易之间，所有的交易情况统计。列表展示股票代码、股票名称、总买入次数、总卖出次数、总交易股数、总买入金额、总卖出金额、总盈利、盈利比例。其中总盈利=总卖出+分红金额-总买入-手续费，盈利比例=总盈利/总买入*100%。当盈利为正数时，使用红色字体，为负数时，使用绿色。总交易股数等于开仓和清仓之间所有买入交易的总股数，总买入次数等于开仓和清仓之间所有买入交易的次数，总卖出次数等于开仓和清仓之间所有卖出交易的次数；总买入金额等于开仓和清仓之间所有买入交易的金额之和；总卖出金额等于开仓和清仓之间所有卖出交易的金额之和；手续费简易计算方法：买入单笔手续费 = MAX(成交金额 × 0.0005, 5元) + 成交金额 × 0.00001，卖出单笔手续费 = MAX(成交金额 × 0.0005, 5元) + 成交金额 × 0.00001 + 成交金额 × 0.0005"

## Clarifications

### Session 2026-04-23

- Q: 股息交易在交易周期中的处理方式 → A: 股息交易包含在周期边界内，但不计入买入/卖出次数和金额统计
- Q: 点击历史记录是否展开显示详细交易明细 → A: 点击展开显示该周期内所有交易明细（日期、类型、价格、数量、手续费）
- Q: 历史记录列表中是否需要显示开仓时间和清仓时间 → A: 列表中显示开仓时间和清仓时间（便于了解交易周期跨度）
- Q: 当股票清仓后再次开仓的周期识别 → A: 清仓后（holding_count=0）的任何买入交易都视为新周期的开始

## User Scenarios & Testing

### User Story 1 - 查看历史开仓记录列表 (Priority: P1)

作为用户，我需要在一个页面上查看我的历史开仓记录列表，以便了解我已完成交易（从买入到清仓）的股票的完整交易统计信息。

**Why this priority**: 这是用户进入该页面的核心需求，需要展示所有已清仓股票的交易统计数据

**Independent Test**: 可以通过打开历史开仓记录页面，验证列表是否正确显示所有已清仓股票的交易统计

**Acceptance Scenarios**:

1. **Given** 用户已打开历史开仓记录页面，**When** 页面加载完成，**Then** 应显示所有已清仓的股票（即从持股数0开始首次买入，到清仓交易之间的完整交易周期），每只股票显示股票代码、股票名称、总买入次数、总卖出次数、总交易股数、总买入金额、总卖出金额、总盈利、盈利比例；当前尚未清仓的股票（即当前 holding_count > 0 的股票）不显示在历史开仓记录中
2. **Given** 历史开仓记录列表已显示，**When** 列表展示时，**Then** 记录应按清仓时间倒序排列（最近清仓的显示在最前面）
3. **Given** 某只股票的交易统计已显示，**When** 总盈利为正数时，**Then** 总盈利和盈利比例应使用红色字体显示
4. **Given** 某只股票的交易统计已显示，**When** 总盈利为负数时，**Then** 总盈利和盈利比例应使用绿色字体显示
5. **Given** 某只股票的交易统计已显示，**When** 总盈利为零时，**Then** 总盈利和盈利比例应使用默认字体颜色显示
6. **Given** 用户已打开历史开仓记录页面，**When** 用户点击刷新按钮，**Then** 应重新加载列表数据并更新显示

### User Story 2 - 计算交易统计数据 (Priority: P1)

作为用户，我需要系统自动计算每只已清仓股票的交易统计数据，以便准确了解我的交易表现。

**Why this priority**: 这是核心计算逻辑，确保数据准确性

**Independent Test**: 可以通过验证交易统计数据的计算结果，确认计算公式正确

**Acceptance Scenarios**:

1. **Given** 系统计算某只已清仓股票的交易统计，**When** 计算总买入次数时，**Then** 应等于开仓和清仓之间所有买入交易（trade_type = 'BUY'）的次数
2. **Given** 系统计算某只已清仓股票的交易统计，**When** 计算总卖出次数时，**Then** 应等于开仓和清仓之间所有卖出交易（trade_type = 'SELL'）的次数
3. **Given** 系统计算某只已清仓股票的交易统计，**When** 计算总交易股数时，**Then** 应等于开仓和清仓之间所有买入交易的股数之和
4. **Given** 系统计算某只已清仓股票的交易统计，**When** 计算总买入金额时，**Then** 应等于开仓和清仓之间所有买入交易的金额之和（成交金额 = 成交价格 × 成交数量）
5. **Given** 系统计算某只已清仓股票的交易统计，**When** 计算总卖出金额时，**Then** 应等于开仓和清仓之间所有卖出交易的金额之和（成交金额 = 成交价格 × 成交数量）
6. **Given** 系统计算某只已清仓股票的交易统计，**When** 计算手续费时，**Then** 应累加所有买入和卖出交易的手续费
7. **Given** 系统计算某只已清仓股票的交易统计，**When** 计算总盈利时，**Then** 总盈利 = 总卖出金额 + 分红金额 - 总买入金额 - 总手续费
8. **Given** 系统计算某只已清仓股票的交易统计，**When** 计算盈利比例时，**Then** 盈利比例 = (总盈利 / 总买入金额) × 100%

---

### User Story 3 - 计算交易手续费 (Priority: P1)

作为用户，我需要系统按照标准费率计算每笔交易的手续费，以便准确计算总盈利。

**Why this priority**: 手续费计算直接影响总盈利的准确性

**Independent Test**: 可以通过手动计算单笔交易手续费，验证系统计算结果

**Acceptance Scenarios**:

1. **Given** 系统计算买入交易的手续费，**When** 计算单笔买入手续费时，**Then** 手续费 = MAX(成交金额 × 0.0005, 5元) + 成交金额 × 0.00001
2. **Given** 系统计算卖出交易的手续费，**When** 计算单笔卖出手续费时，**Then** 手续费 = MAX(成交金额 × 0.0005, 5元) + 成交金额 × 0.00001 + 成交金额 × 0.0005
3. **Given** 系统计算某只股票的总手续费，**When** 计算总手续费时，**Then** 应等于开仓和清仓之间所有买入和卖出交易手续费的总和

---

### User Story 4 - 识别开仓和清仓周期 (Priority: P1)

作为用户，我需要系统正确识别每只股票的开仓和清仓周期，以便准确统计该周期内的交易数据。

**Why this priority**: 开仓和清仓周期的识别是统计的基础，直接影响统计范围

**Independent Test**: 可以通过检查交易记录，验证开仓和清仓周期的识别是否正确

**Acceptance Scenarios**:

1. **Given** 系统识别某只股票的开仓和清仓周期，**When** 查找开仓点时，**Then** 应找到该股票持股数（holding_count）从0开始首次买入的交易记录
2. **Given** 系统识别某只股票的开仓和清仓周期，**When** 查找清仓点时，**Then** 应找到该股票持股数（holding_count）变为0的卖出交易记录
3. **Given** 系统识别某只股票的开仓和清仓周期，**When** 统计交易数据时，**Then** 应只统计开仓和清仓之间（包含开仓和清仓交易）的所有交易记录
4. **Given** 某只股票存在多个开仓和清仓周期，**When** 显示历史记录时，**Then** 每个开仓和清仓周期应作为一条独立的记录显示
5. **Given** 某只股票清仓后（holding_count=0）又有新的买入交易，**When** 识别新周期时，**Then** 应将该买入交易视为新周期的开仓点，无论与上次清仓的时间间隔

---

### User Story 5 - 查看交易明细 (Priority: P2)

作为用户，我需要点击某条历史记录展开查看该周期内的所有交易明细，以便验证统计数据并了解详细的交易过程。

**Why this priority**: 提供交易明细的可追溯性，增强用户对统计数据的信任

**Independent Test**: 可以通过点击历史记录，验证是否正确展开显示该周期内的所有交易明细

**Acceptance Scenarios**:

1. **Given** 用户已打开历史开仓记录页面，**When** 用户点击某条历史记录，**Then** 应展开显示该周期内所有交易明细，包括交易日期、交易类型、交易价格、交易数量、单笔手续费
2. **Given** 用户已展开某条历史记录的交易明细，**When** 用户再次点击该记录，**Then** 应收起交易明细列表
3. **Given** 用户已展开某条历史记录的交易明细，**When** 用户点击另一条历史记录，**Then** 原记录应收起，新记录的交易明细应展开
4. **Given** 交易明细列表已展开，**When** 列表展示时，**Then** 交易记录应按交易时间正序排列（从开仓到清仓的时间顺序）

## Requirements

### Functional Requirements

- **FR-001**: 系统 MUST 从 trade_record 表查询所有已清仓股票的交易记录
- **FR-001.1**: 系统 MUST 提供刷新按钮，点击后重新加载列表数据
- **FR-002**: 系统 MUST 识别每只股票的开仓点（holding_count 从 0 开始首次买入）和清仓点（holding_count 变为 0）
- **FR-003**: 系统 MUST 将每个开仓和清仓周期作为一条独立的历史记录
- **FR-004**: 系统 MUST 计算每个周期的总买入次数（trade_type = 'BUY' 的交易次数）
- **FR-005**: 系统 MUST 计算每个周期的总卖出次数（trade_type = 'SELL' 的交易次数）
- **FR-005.1**: 系统 MUST 在统计买入/卖出次数和金额时，排除股息交易（trade_type = 'DIVIDEND'），但股息交易仍包含在周期边界内
- **FR-006**: 系统 MUST 计算每个周期的总交易股数（所有买入交易的 trade_count 之和）
- **FR-007**: 系统 MUST 计算每个周期的总买入金额（所有买入交易的 trade_price × trade_count 之和）
- **FR-008**: 系统 MUST 计算每个周期的总卖出金额（所有卖出交易的 trade_price × trade_count 之和）
- **FR-009**: 系统 MUST 计算每个周期的总手续费（所有买入和卖出交易手续费之和）
- **FR-010**: 系统 MUST 按照以下公式计算买入交易手续费：MAX(成交金额 × 0.0005, 5元) + 成交金额 × 0.00001
- **FR-011**: 系统 MUST 按照以下公式计算卖出交易手续费：MAX(成交金额 × 0.0005, 5元) + 成交金额 × 0.00001 + 成交金额 × 0.0005
- **FR-012**: 系统 MUST 按照以下公式计算总盈利：总盈利 = 总卖出金额 + 分红金额 - 总买入金额 - 总手续费
- **FR-013**: 系统 MUST 按照以下公式计算盈利比例：盈利比例 = (总盈利 / 总买入金额) × 100%
- **FR-014**: 系统 MUST 在总盈利为正数时，使用红色字体显示总盈利和盈利比例
- **FR-015**: 系统 MUST 在总盈利为负数时，使用绿色字体显示总盈利和盈利比例
- **FR-016**: 系统 MUST 按清仓时间倒序排列历史记录（最近清仓的显示在最前面）
- **FR-017**: 系统 MUST 在列表中显示以下字段：股票代码、股票名称、开仓时间、清仓时间、总买入次数、总卖出次数、总交易股数、总买入金额、总卖出金额、分红金额、总盈利、盈利比例
- **FR-017.1**: 系统 MUST 计算每个周期的分红金额（所有股息交易的 trade_price × holding_count × 0.9 之和）
- **FR-018**: 系统 MUST 在用户点击历史记录时展开/收起该周期内的交易明细列表
- **FR-019**: 系统 MUST 只同时展开一条历史记录的交易明细，点击另一条时自动收起前一条
- **FR-020**: 系统 MUST 在交易明细列表中显示以下字段：交易日期、交易类型、交易价格、交易数量、单笔手续费
- **FR-021**: 系统 MUST 按交易时间正序排列交易明细（从开仓到清仓的时间顺序）

### Key Entities

- **历史开仓记录**: 代表一只股票从开仓到清仓的完整交易周期，包含属性：股票代码、股票名称、总买入次数、总卖出次数、总交易股数、总买入金额、总卖出金额、分红金额、总盈利、盈利比例、开仓时间、清仓时间
- **交易周期**: 代表从持股数0开始首次买入（开仓）到持股数变为0（清仓）之间的所有交易记录集合

### Display Conventions

- **盈利颜色**: 总盈利和盈利比例为正数（盈利）时显示红色，为负数（亏损）时显示绿色，为零时显示默认颜色
- **交易类型颜色**: 买入（BUY）显示绿色，卖出（SELL）显示红色，股息（DIVIDEND）显示橙色
- **成交金额符号**: 买入（BUY）显示负数（资金流出），卖出（SELL）和股息（DIVIDEND）显示正数（资金流入）
- **股息成交金额**: 股息交易的成交金额 = trade_price × holding_count × 0.9
- **金额格式**: 金额显示保留两位小数（如：1234.56）
- **比例格式**: 盈利比例显示保留两位小数，并添加百分号（如：12.34%）
- **日期格式**: 开仓时间和清仓时间只显示到天（忽略时分秒）

## Success Criteria

### Measurable Outcomes

- **SC-001**: 用户可在3秒内打开历史开仓记录页面并看到记录列表
- **SC-002**: 交易统计数据计算准确率达到100%
- **SC-003**: 手续费计算准确率达到100%
- **SC-004**: 开仓和清仓周期识别准确率达到100%

## Assumptions

- trade_record 表已存在且包含完整的交易记录
- holding_count 字段正确记录了每笔交易后的持仓数量
- 系统支持买入（BUY）、卖出（SELL）和股息（DIVIDEND）三种交易类型
- 股息交易包含在开仓和清仓周期边界内，但不计入买入/卖出次数和金额统计
- 每只股票可能存在多个开仓和清仓周期，每个周期应独立统计
- 清仓定义为 holding_count 变为 0 的卖出交易
- 开仓定义为 holding_count 从 0 开始的首次买入交易
