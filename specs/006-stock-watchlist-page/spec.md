# Feature Specification: 自选股页面

**Feature Branch**: `006-stock-watchlist-page`
**Created**: 2026-04-04
**Status**: Draft
**Input**: User description: "实现自选股页面， - 页面展示股票列表； - 可以添加或删除列表项； - 可以设置股票的买入阈值和卖出阈值 - 可以开启或关闭某只股票的监控 - 启动程序后，开启定时任务，对开启监控的股票，定时通过设置页面设置的API，获取股票价格，定时任务时间间隔使用设置页面设置的轮询间隔； - 关闭程序时，停止定时任务； 约束条件： - 卖出阈值必须高于买入阈值；"

## 需求变更记录



### 变更 2026-04-10

- **变更内容**: 定时任务获取到最新价格后，不再保存到数据库；数据库表中删除当前价格字段；页面上展示的当前价格直接使用API实时获取
- **变更原因**: 当前价格是实时数据，无需持久化，直接从API获取更能保证数据时效性

### 变更 2026-04-16

- **变更内容**: 自选股列表在"名称"列与"当前价格"列之间增加五列：开盘价、当日最高价、当日最低价、涨跌额、涨跌幅。涨跌额=当前价格-昨日收盘价；涨跌幅=(当前价格-昨日收盘价)/昨日收盘价×100%，保留两位小数。涨跌额/涨跌幅为正时红色显示，为负时绿色显示（遵循A股红涨绿跌惯例）
- **变更原因**: 用户需要更完整的日内行情信息，便于快速判断股票当日走势和波动范围，无需切换到外部行情软件

## Clarifications

### Session 2026-04-04

- Q: 当股票价格触发阈值条件时，用户通过什么方式收到告警通知？ → A: 同时使用页面显示和系统通知
- Q: 自选股列表数据和监控配置数据是否需要持久化保存？ → A: 需要持久化（程序退出后重启数据仍保留）
- Q: 自选股列表数据和监控配置数据需要保存到哪里？ → A: 需要保存到数据库中
- Q: 用户是否需要能够查看历史告警记录？ → A: 不需要，告警仅即时通知，历史不保存
- Q: 当某只股票触发买入或卖出告警后，该股票的监控开关状态是否需要自动关闭？ → A: 保持当前监控状态不变
- Q: 当股票触发告警后，是否需要防止在一定时间内对该股票重复触发相同类型的告警？ → A: 需要，相同类型告警在一定时间内只触发一次（如5分钟内）

## User Scenarios & Testing

### User Story 1 - 查看和管理自选股列表 (Priority: P1)

作为用户，我需要在一个页面上查看我关注的股票列表，以便快速了解我关注股票的当前状态。

**Why this priority**: 这是核心功能，用户进入自选股页面的首要目的就是查看股票列表

**Independent Test**: 可以通过打开自选股页面，验证列表是否正确显示所有已添加的股票，并能展示每只股票的基本信息

**Acceptance Scenarios**:

1. **Given** 用户已打开自选股页面，**When** 页面加载完成，**Then** 应显示用户之前添加的所有股票，列表中每只股票应显示股票代码、名称、开盘价、当日最高价、当日最低价、涨跌额、涨跌幅、当前价格、买入阈值、卖出阈值和监控状态
2. **Given** 用户已打开自选股页面且列表不为空，**When** 用户查看列表，**Then** 每只股票的监控开关状态应清晰标识（开启/关闭）
3. **Given** 用户已打开自选股页面且列表不为空，**When** 用户查看列表，**Then** 已开启监控的股票应排在未开启监控的股票上方
4. **Given** 用户已打开自选股页面，**When** 页面加载完成，**Then** 应在页面顶部显示统一的最后成功刷新时间

---

### User Story 2 - 添加和删除股票 (Priority: P1)

作为用户，我需要向自选股列表添加新股票或删除不再需要的股票，以便管理我的投资关注列表。

**Why this priority**: 管理和维护自选股列表是基础功能，没有此功能用户无法进行任何列表操作

**Independent Test**: 可以通过添加一只股票验证列表更新，然后删除同一只股票验证列表恢复原状

**Acceptance Scenarios**:

1. **Given** 用户已打开自选股页面，**When** 用户点击添加股票按钮并完成添加操作，**Then** 新股票应出现在列表中，且监控开关状态默认为关闭
2. **Given** 用户已打开自选股页面且列表中已有股票，**When** 用户选择删除某只股票并确认删除，**Then** 该股票应从列表中移除
3. **Given** 用户已打开自选股页面，**When** 用户尝试添加一只已存在的股票，**Then** 系统应提示该股票已在列表中，不允许重复添加

---

### User Story 3 - 手动刷新股票价格 (Priority: P2)

作为用户，我需要手动刷新已开启监控的股票价格，以便在定时任务间隔期间主动获取最新价格。

**Why this priority**: 用户可能不想等待定时任务，想要立即获取最新价格

**Independent Test**: 可以通过点击刷新按钮，验证已开启监控的股票价格被更新

**Acceptance Scenarios**:

1. **Given** 用户已打开自选股页面且列表中有已开启监控的股票，**When** 用户点击刷新按钮，**Then** 系统应立即使用单一定时任务获取所有已开启监控股票的最新价格
2. **Given** 用户已打开自选股页面，**When** 用户点击刷新按钮，**Then** 页面应显示刷新操作正在执行的状态
3. **Given** 用户已打开自选股页面，**When** 刷新过程中，**Then** 已关闭监控的股票不应被获取价格

---

### User Story 4 - 设置股票阈值和监控开关 (Priority: P1)

作为用户，我需要为每只股票设置买入阈值、卖出阈值，以及开启或关闭该股票的监控，以便实现个性化的股票监控策略。

**Why this priority**: 阈值设置和监控开关是实现自动化监控的核心配置，直接影响功能价值

**Independent Test**: 可以通过为一股票设置阈值、开启监控，验证设置被正确保存；关闭监控后验证该股票不在定时任务中获取价格

**Acceptance Scenarios**:

1. **Given** 用户已打开自选股页面，**When** 用户为某只股票设置买入阈值和卖出阈值并保存，**Then** 阈值应被正确保存，且卖出阈值必须高于买入阈值的约束应被系统强制校验
2. **Given** 用户已打开自选股页面，**When** 用户尝试将卖出阈值设置为低于或等于买入阈值，**Then** 系统应拒绝保存并提示错误信息"卖出阈值必须高于买入阈值"
3. **Given** 用户已打开自选股页面，**When** 用户开启某只股票的监控开关，**Then** 该股票应被标记为已开启监控状态，列表应自动将该股票排序到未开启监控的股票上方
4. **Given** 用户已打开自选股页面，**When** 用户关闭某只股票的监控开关，**Then** 该股票应被标记为已关闭监控状态，列表应自动将该股票排序到未开启监控的股票下方

---

### User Story 5 - 自动监控股票价格 (Priority: P1)

作为用户，我希望程序启动后自动开始监控已开启监控的股票，通过定时任务定期获取股票价格，以便及时掌握股票动态。

**Why this priority**: 这是自动化的核心功能，确保用户无需手动操作即可持续监控股票

**Independent Test**: 启动程序后，验证定时任务对开启监控的股票按设定间隔获取价格；关闭程序时验证定时任务被停止

**Acceptance Scenarios**:

1. **Given** 程序已启动且存在已开启监控的股票，**When** 系统运行中，**Then** 定时任务应按设置页面配置的轮询间隔，定期获取这些股票的当前价格
2. **Given** 程序已启动且某只股票已开启监控，**When** 定时任务获取到新价格时，**Then** 系统应比较价格与设置的阈值，判断是否触发告警条件
3. **Given** 程序正在运行且定时任务正在执行，**When** 用户关闭程序，**Then** 定时任务应被正确停止，不再继续获取股票价格
4. **Given** 程序已启动，**When** 程序运行期间，**Then** 已关闭监控的股票不应参与定时价格获取

---

### Edge Cases

- 当自选股列表为空时，页面应显示友好提示引导用户添加第一只股票
- 当API获取股票价格失败时，系统应记录错误并继续监控其他股票，不应影响整体定时任务
- 当设置页面未配置API或轮询间隔时，系统应有默认值或提示用户配置
- 当股票价格获取后达到或超过卖出阈值，或达到或低于买入阈值时，应同时通过页面显示和系统通知方式触发告警
- 当用户快速多次点击添加股票时，系统应防止重复提交
- 当股票触发告警后，5分钟内相同类型的告警不再重复触发

## Requirements

### Functional Requirements

- **FR-001**: 系统 MUST 显示自选股列表，列表项包含股票代码、名称、开盘价、当日最高价、当日最低价、涨跌额、涨跌幅、当前价格、买入阈值、卖出阈值、监控状态；开盘价、最高价、最低价、涨跌额、涨跌幅和当前价格通过API实时获取，不持久化到数据库；涨跌额=当前价格-昨日收盘价，涨跌幅=(当前价格-昨日收盘价)/昨日收盘价×100%（保留两位小数）；涨跌额和涨跌幅为正时红色显示，为负时绿色显示
- **FR-002**: 用户 MUST 能够添加新股票到自选股列表
- **FR-003**: 用户 MUST 能够从自选股列表中删除已有股票
- **FR-004**: 系统 MUST 不允许添加重复的股票到列表
- **FR-005**: 用户 MUST 能够为每只股票设置买入阈值和卖出阈值
- **FR-006**: 系统 MUST 强制校验卖出阈值必须高于买入阈值，不满足时拒绝保存并提示错误
- **FR-007**: 用户 MUST 能够开启或关闭单只股票的监控功能
- **FR-008**: 用户 MUST 能够手动触发一次股票价格刷新
- **FR-009**: 系统 MUST 使用单一HTTP请求，一次获取所有已开启监控股票的当前价格
- **FR-010**: 系统 MUST 使用设置页面配置的API地址和轮询间隔进行股票价格获取
- **FR-011**: 系统 MUST 在程序关闭时自动停止定时任务
- **FR-012**: 系统 MUST 当股票价格达到或超过卖出阈值时触发卖出告警，告警同时通过页面显示和系统通知方式提醒用户
- **FR-013**: 系统 MUST 当股票价格达到或低于买入阈值时触发买入告警，告警同时通过页面显示和系统通知方式提醒用户
- **FR-014**: 系统 MUST 将自选股列表数据和监控配置数据持久化保存到数据库，确保程序重启后数据不丢失
- **FR-015**: 系统 MUST 对相同类型的告警在5分钟内防止重复触发（防抖机制）
- **FR-016**: 系统 MUST 在页面顶部显示统一的最后成功刷新时间
- **FR-017**: 系统 MUST 自动按监控状态排序股票列表，已开启监控的股票排在未开启监控的股票上方
- **FR-018**: 系统 MUST 定时任务只在交易时间段内调用远程API获取股票价格，非交易时间段内不进行价格获取
- **FR-019**: 系统 MUST 从config表中读取API地址、轮询间隔和交易时间段配置
- **FR-020**: 系统 MUST 当价格高于卖出阈值时，使用红色字体颜色显示价格；当价格低于买入阈值时，使用绿色字体颜色显示价格

### Key Entities

- **自选股**: 代表用户关注的单只股票，包含属性：股票代码、名称、买入阈值、卖出阈值、监控开关状态（默认为关闭）
- **股票价格**: 代表某只股票的实时价格数据，包含属性：股票标识、当前价格、开盘价、昨日收盘价、当日最高价、当日最低价、涨跌额（当前价格-昨日收盘价）、涨跌幅（(当前价格-昨日收盘价)/昨日收盘价×100%）、获取时间戳
- **监控配置**: 代表整体监控设置，包含属性：API地址、轮询间隔（分钟）、上午交易时间段、下午交易时间段

## Success Criteria

### Measurable Outcomes

- **SC-001**: 用户可在3秒内打开自选股页面并看到已添加的股票列表
- **SC-002**: 用户添加一只新股票到列表的操作在2秒内完成
- **SC-003**: 定时任务按设定的轮询间隔获取股票价格，间隔误差不超过±5秒
- **SC-004**: 程序关闭时，定时任务在1秒内被完全停止
- **SC-005**: 系统正确校验所有阈值设置，100%防止卖出阈值低于或等于买入阈值的情况保存成功
- **SC-006**: 用户可以随时开启或关闭任一股票的监控，开关操作立即生效
- **SC-007**: 当股票价格触发阈值条件时，告警在下一价格获取周期内被触发
- **SC-008**: 同一只股票的相同类型告警在5分钟内不会重复触发

## Assumptions

- 用户已在设置页面配置了有效的股票价格获取API地址
- 用户已在设置页面配置了合理的轮询间隔（建议默认1分钟）
- 用户已在设置页面配置了交易时间段（默认上午09:30-11:30，下午13:00-15:00）
- 股票价格API返回的数据格式包含股票标识和价格信息
- 程序关闭指的是正常关闭流程，异常退出情况下定时任务可能无法被停止
- 单次API调用失败不影响后续定时任务执行，系统会持续重试
